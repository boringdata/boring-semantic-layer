### What works

- `src/boring_semantic_layer/calc_analyzer.py`
  - `analyze_calc_expr(expr, known_measures, base_table_op) -> CalcExprAnalysis`
  - Classification record: `pushable`, `references_AllOf`, `has_window`, `post_agg_only`, `depends_on`, `inline_aggs`
  - Distinguishes empty-window-over-reduction (the `t.all(...)` totals pattern) from real partitioned/ordered windows (moving avg, rank).
  - 8 unit tests in `tests/test_calc_analyzer.py`.

- `src/boring_semantic_layer/calc_compiler.py`
  - `IbisCalcScope` — dual-table dispatch. `t.measure_name` returns a column on a virtual aggregated table; `t.column_name` returns the base-table column (so inline `t.distance.sum()` continues to work).
  - `evaluate_calc_lambda(fn, base_tbl, known_measures)` — runs the lambda against `IbisCalcScope`, returns `(expr, virtual_agg_tbl)`.
  - `classify_calc_lambda(...)` — evaluates the lambda then runs the analyzer.
  - `compile_calc_measure(expr, virtual_agg_tbl, real_agg_tbl)` — substitutes the virtual table for the real one via `op.replace(...)`.
  - `compile_calc_measures(real_agg_tbl, calc_exprs)` — applies all calcs via a single `mutate(...)`.
  - 9 unit tests in `tests/test_calc_compiler.py`, including end-to-end `pct = flight_count / t.all(flight_count)` against duckdb (sums to 1.0 ✓).

## Design decisions worth re-checking

1. **`t.all(...)` semantics for non-sum measures.** The new compiler emits `column.sum().over(window())` — the *windowed sum of the post-agg column*. For sum-style measures this matches the unfiltered total; for `mean`-style measures it does not (sum of per-group means ≠ overall mean). The existing curated-AST path computes a true totals table by **re-aggregating on the unfiltered base table**, which is more correct. Decide whether v1 of the analyzer-based path should preserve that behavior. If yes, the compiler needs a totals-table cross-join branch when `analysis.references_AllOf and depends_on intersects {known measures}`. **Recommendation:** add this branch in Step 2 to avoid behavior regression.

2. **Calc-of-calc references.** Today's `compile_grouped_with_all` adds all calcs in a single `mutate(...)`, so calc-of-calc doesn't work. The new compiler inherits this. If we want calc-of-calc support, sort calcs topologically by `analysis.depends_on` and apply in dependency order. Probably out of scope for this branch; flag and ship.

3. **Hashability of stored ibis expressions.** Calc measures get stored on `SemanticTableOp.calc_measures` which is a `FrozenDict`. Ibis expressions are hashable, but `(expr, virtual_agg_tbl)` tuples need to round-trip through `to_tagged`/`from_tagged`. Confirm in Step 6.
