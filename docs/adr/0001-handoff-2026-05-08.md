### What's now wired up

- `src/boring_semantic_layer/calc_analyzer.py` walks the ibis tree but
  no longer descends through `Relation` nodes — base-table window
  expressions in the source no longer pollute the classification of
  calc measures defined on top.
- `src/boring_semantic_layer/calc_compiler.py` adds:
  - `IbisCalcScope` with column-first dispatch, suffix-resolution for
    prefixed measure names on joined models, and `UnknownMeasureRefError`
    on unknown lookups.
  - `apply_calc_measures(real_agg_tbl, base_tbl, calc_lambdas, known)`
    — the query-time entrypoint. Topologically orders calc-of-calc
    chains and applies them via successive `mutate` calls.
  - `rename_measure_refs(expr, vt, name_map)` — kept for completeness
    but no longer needed at runtime since the lambda re-runs against a
    prefix-aware `IbisCalcScope` at query time.
- `src/boring_semantic_layer/ops.py`:
  - `CalcMeasure` is the new storage shape on `SemanticTableOp.calc_measures`
    (a frozen dataclass with `expr`, `description`, `requires_unnest`,
    `depends_on`, `metadata`). Curated AST helpers (`_is_calculated_measure`,
    `_matches_aggregation_pattern`, `_find_matching_measure`,
    `_resolve_aggregation_exprs`, `_expand_calc_measure_refs`,
    `_collect_measure_refs`) are removed.
  - `_classify_measure` runs the lambda once against `IbisCalcScope`,
    walks the result with `analyze_calc_expr`, and routes pushable
    expressions to `_make_base_measure` and post-aggregation expressions
    to `CalcMeasure`.
  - `_build_aggregation_plan` / `_compile_aggregation` replace the curated
    `compile_grouped_with_all` pipeline. Bare-name lambdas from
    `aggregate("name")` are detected directly; calc dependencies are
    walked transitively (with suffix matching on prefixed names) so the
    base aggregation produces every column the calc lambdas read.
  - The pre-agg path's `_apply_calc_specs` and `_to_untagged_with_deferred_joins`
    both route through `apply_calc_measures` now; nothing in the planner
    branches on AST node types anymore.
  - `_update_measure_refs_in_calc` is a no-op: `IbisCalcScope`'s
    suffix-resolution handles prefix bridging at query time.
- `src/boring_semantic_layer/measure_scope.py` is reduced to the
  pass-through `MeasureScope` / `ColumnScope` proxies still used by
  `SemanticMutateOp` (which survives this branch). All curated AST
  classes (`MeasureRef`, `AllOf`, `BinOp`, `MethodCall`,
  `AggregationExpr`, `MeasureExpr`, `_Node`, `_PendingMethodCall`,
  `validate_calc_ast`, `DeferredColumn`) are deleted.
- `src/boring_semantic_layer/compile_all.py` is **deleted**. The nested-array
  helpers it owned (`_join_tables`, `_build_session_table`,
  `_build_nested_level_table`, `_build_level_aggregations`, etc.)
  moved to `nested_compile.py`. `_get_ibis_module` moved alongside as
  `nested_compile.get_ibis_module`.
- `src/boring_semantic_layer/serialization/extract.py`:
  - `serialize_calc_measures` walks each `CalcMeasure.expr` callable
    via `expr_to_structured` and stores the resolver tree plus
    `description`, `requires_unnest`, and `depends_on`. No more
    AST-tag dispatch.
  - `deserialize_calc_measures` reconstructs `CalcMeasure(expr=Deferred(...), depends_on=...)`
    and lets the planner re-run the lambda at query time.
  - `utils.serialize_resolver` / `deserialize_resolver` now handle
    `Item` resolvers (needed for `t["prefixed.name"]` patterns).

## Known limitations / follow-ups

1. **`t.all(...)` over non-sum measures.** Resolved. ``IbisCalcScope``
   now routes ``t.all(measure_ref)`` to a parallel totals virtual
   table; ``apply_calc_measures`` (and the ``_compile_aggregation``
   path) build a real totals aggregation by re-running the agg specs
   on ``base_tbl`` without group keys, apply non-AllOf calc measures
   to it, then cross-join it into the per-group result with prefixed
   column names. Calc compilation rewrites ``Field(totals_vt, name)``
   references to point at the prefixed columns. Non-sum chains
   (``avg_distance / t.all(avg_distance)``) now match the
   curated-AST behavior — the formula is recomputed on the totals
   aggregation rather than summed across per-group results. See
   ``test_all_of_non_sum_measure_uses_totals_table`` and
   ``test_all_of_multilayer_calc_measure``.
2. **Inline aggregations inside `t.all(...)` are now supported.**
   Patterns like `t.value.sum() / t.all(t.value.sum())` and
   `t.value.mean() / t.all(t.value.mean())` work end-to-end via the
   inline-reduction lift in ``calc_compiler.lift_inline_reductions``:
   each unique reduction over the base table is named, added to both
   the per-group base aggregation and the totals aggregation, and the
   calc expression is rewritten so bare reductions become
   ``Field(vt, anon_name)`` while window-wrapped reductions (the
   ``t.all(...)`` shape) become ``Field(totals_vt, anon_name)``.
3. **Calc-of-calc** is now supported via topological ordering inside
   `apply_calc_measures` (each calc is added to the result via its own
   `mutate(...)` so subsequent calcs see it as a column).
4. **`_update_measure_refs_in_calc`** is a no-op stub. The call site in
   `_merge_fields_with_prefixing` can be removed once we're confident
   nothing third-party imports it.
5. **`SemanticMutateOp`** still exists per the ADR scope (Phase 1+2,
   not Phase 3). Phase 3 (drop the chained `.mutate()` operator) would
   be a separate ADR / branch.

## Quick reference for future contributors

- A calc measure's `depends_on` is captured **at classification time**
  by walking the ibis tree the lambda produced when run against
  `IbisCalcScope`. The set is preserved through serialization so the
  query planner can auto-include base-measure dependencies even on a
  reconstructed model.
- Calc measures are stored as `CalcMeasure(expr=callable, depends_on=...)`.
  At query time the planner re-runs `expr` against a fresh
  `IbisCalcScope` (over the actual filtered base table) and substitutes
  the virtual aggregated table for the real one. This avoids embedding
  ibis-table identity into the stored lambda.
- For prefix bridging on joined models, the `IbisCalcScope` does
  unique-suffix matching: `t.flight_count` resolves to
  `flights.flight_count` when there's exactly one such suffix in the
  known-measure set.
